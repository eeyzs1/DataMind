## 1. 成本设计理念：弹性成本与弹性代码同源

### 1.1 核心承诺

DataMind的成本模型与弹性框架深度绑定。**同样的业务代码，在不同的profile下产生完全不同的成本，成本的增加对应的是能力的增加，而不是代码复杂度的增加。**

```
同一个 dbt 模型文件 fct_daily_revenue.sql:

  dev profile → DuckDB 本地执行 → $0/月
  prod profile → Spark 集群执行 → $3,000/月

代码差异：0行
成本差异：$3,000/月
能力差异：从单机100万行 → 集群10亿行
```

### 1.2 成本弹性原则

| 原则 | 含义 | 实现 |
|------|------|------|
| **代码零改动** | profile升级不改变业务代码 | 接口-后端模式，只换后端实现 |
| **成本按需增长** | 能力越大成本越高，但linearly，非exponentially | 按profile渐进 |
| **成本可见** | 每个操作的成本实时可见 | `profile.yaml` 一目了然 |
| **预算硬顶** | 每级profile有预算上限 | prod+有预算控制 |

## 5. 成本优化策略（所有profile适用）

### 5.1 成本控制金字塔

```
                    ┌──────┐
                    │ 按需  │  弹性计算：Spot实例/预留实例混合
                   ┌┴──────┴┐
                   │ 冷热分离 │  冷数据自动降级存储（S3 Intelligent-Tiering）
                  ┌┴────────┴┐
                  │ 计算下推   │  谓词下推+列裁剪，减少IO
                 ┌┴──────────┴┐
                 │ 数据压缩    │  Parquet+ZSTD，压缩比5:1-10:1
                ┌┴────────────┴┐
                │ 预算硬顶      │  profile级别预算上限
               ┌┴──────────────┴┐
               │ 成本可视化      │  每个查询/任务的成本估算
               └────────────────┘
```

### 5.2 优化措施（profile适用性）

| 措施 | 预期节省 | dev | staging | prod | global |
|------|---------|-----|---------|------|--------|
| Parquet+ZSTD压缩 | 存储降低60% | ✅ 自动 | ✅ 自动 | ✅ 自动 | ✅ 自动 |
| 谓词下推+列裁剪 | 计算减少50% | ✅ DuckDB自动 | ✅ DuckDB自动 | ✅ Spark自动 | ✅ Spark自动 |
| 冷数据归档 | 冷存储降低70% | ❌ | ❌ | ✅ S3 Glacier | ✅ 多区域Glacier |
| Spot/预留实例混合 | 计算降低40-60% | ❌ | ❌ | ✅ | ✅ |

### 6.1 价值量化

| 价值维度 | dev→staging | staging→prod | prod→global |

| 首先解决的问题 | 概念验证 | 团队协作效率 | 大数据处理能力 | 全球合规 |

| 升级 | 触发条件 | 投资理由 |

## 7. Profile切换成本

### 7.1 一次性切换成本

从当前profile升级到下一个profile，需要的一次性投入：

| 升级路径 | 基础设施搭建 | 配置修改 | 数据迁移 | 预计时间 |
|---------|-------------|---------|---------|---------|
| dev→staging | 部署MinIO、Dagster、Keycloak、Superset | 修改 `profile.yaml` | 本地Parquet同步到MinIO | 1-2天 |
| staging→prod | 部署Spark集群、Kafka、S3/Iceberg、OpenMetadata | 修改 `profile.yaml` | MinIO数据迁移到S3，格式转为Iceberg | 1-2周 |
| prod→global | 部署额外区域集群、全局控制平面、CDN | 修改 `profile.yaml`，添加区域配置 | 区域间元数据同步 | 2-4周 |

### 7.2 零代码改动保证

```yaml
# profile升级只需要改这一个文件

# staging 的配置:
profile: staging
storage:
  backend: minio
  endpoint: http://minio.internal:9000
compute:
  backend: duckdb
scheduler:
  backend: dagster

# 升级到 prod 只需要改为:
profile: prod
storage:
  backend: s3
  bucket: datamind-prod-eu
compute:
  backend: spark
  master: spark://prod-master:7077
scheduler:
  backend: dagster_cluster
```

**所有 dbt 模型、Python ETL 脚本、API 代码、质量规则，完全不变。**

## 8. 每月LLM成本（独立于profile的附加成本）

LLM成本不随profile变化，随使用量变化：

| LLM场景 | 调用频率 | 单次成本 | 月度成本 |
|---------|---------|---------|---------|
| Text-to-SQL | ~100次/天 | ~$0.01 | ~$30 |
| 异常解释 | ~10次/天 | ~$0.02 | ~$6 |
| 元数据描述 | ~50次/月(增量) | ~$0.01 | ~$0.5 |
| 代码生成 | ~20次/天 | ~$0.02 | ~$12 |
| RAG分析（global专属） | ~50次/天 | ~$0.05 | ~$75 |
| **总计** | | | **staging: ~$48/月, prod: ~$48/月, global: ~$123/月** |

LLM成本在各profile中都很低（<5%的infra成本），因为严格遵循"LLM不碰数据行"原则。详见 [11-ai-integration.md](11-ai-integration.md)。
